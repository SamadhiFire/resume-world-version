## Messenger (abeto.co) 原团队技术分享 — 要点总结

本文整理自 Messenger 原开发团队的技术分享，涵盖他们在 WebGL 浏览器游戏中所做的关键技术决策。这些方案对"个人作品集世界"项目有直接参考价值。

---

### 核心设计动机：平静但有一点奇怪

Messenger 的出发点不是做一个复杂的大型游戏，而是捕捉一种"calm but weird"的氛围：整体很安静、放松，但世界里又有一点点不寻常的陌生感。团队参考了亚洲中型城市、郊区、海边、寺庙、工厂等空间，把城市混凝土和自然环境混在一起，形成一种既真实又轻微异化的探索感。

**对我们的启示**：个人作品集世界也不应该只是"把简历放进 3D 场景"。它需要一个清晰的情绪基调，例如"轻松探索一个人的经历地图"。世界可以很小，但要有记忆点：比如实习大道偏城市/办公氛围，Vibe Coding 工坊偏工具与机械感，户外探索径偏自然与照片回忆。每个区域都服务于"认识这个人"这个目标。

### 微型星球结构

他们一开始就把世界设想成一个很小的球形星球，玩家可以朝任意方向走，最后回到起点。这带来了三个关键挑战：动态相机、物体朝向、中心重力。所有树、文字、角色都要在曲面上自然站立，摄像机也要适应玩家处在任意角度的状态。

**对我们的启示**：球形世界很有记忆点，但第一版不一定要直接做完整球面物理。更稳妥的 MVP 是先做一个"视觉上像微型世界"的平面/低曲率岛屿：中央路口 + 1 条完整主题路线。等移动、镜头、内容弹窗、性能都稳定后，再考虑把地形升级成球面或分区小星球。这样项目更容易真正跑起来。

### 世界建模方法：先易于编辑，再追求形态

团队尝试过两种方式来制作球形世界：一种是把普通资产程序化变形到球面上，另一种是先按"展开的立方体"建模，再把它转成球。最后他们选择第二种，因为这样更容易调整世界设计和摆放细节。

**对我们的启示**：本项目不应一开始就追求复杂程序化生成。更适合采用"模块化场景块"：道路块、建筑块、展示牌、照片节点、作品终端、山路节点等都做成可复用组件。先保证内容好摆、好改、好扩展，再慢慢提升视觉表现。

### 美术方向：手绘感、描边和低成本细节

Messenger 的视觉不是高写实，而是有一点不完美的手绘感。团队用自定义描边系统控制线条的粗细、颜色和透明度，让低模几何体也能有清晰轮廓和丰富细节。它的重点不是模型有多复杂，而是轮廓、色彩和局部小物件一起营造世界感。

**对我们的启示**：个人项目更应该走风格化路线，而不是写实路线。可以优先采用 low-poly + toon/outline shader + 小色板的方案。这样既节省资产制作成本，也更利于移动端运行。HR 看的是信息和记忆点，不是 3A 级模型精度。

### 区域设计：七个有辨识度的小地点

Messenger 的小星球包含多个有差异的地点，例如街区、广场、墓地、海滩、山中寺庙、森林、工厂。每个地点都不大，但有独立气质和可识别物件。世界因此显得丰富，而不是靠扩大地图面积来制造内容量。

**对我们的启示**：我们的五个内容板块也应该做成"小而明确"的地点，而不是一大片平均铺开的信息区：

- 实习经历：城市街区/办公大道，用建筑、招牌、数据看板承载经历。
- Vibe Coding 作品：工坊/控制室，用终端、机器、工具台展示项目。
- AIGC 作品：小影院/影像展厅，用屏幕和灯光营造视觉内容。
- 兴趣爱好：山路/营地/路线牌，用照片、地图、路径节点承载户外经历。
- 个人档案：中央服务站/档案馆，放教育、技能、联系方式和快速简历入口。

第一版只需要把其中一个区域做完整，其他区域用路牌或未开放入口占位即可。

### 任务设计：简单、低压力、有理由行走

Messenger 最后选择了最简单的故事：送信。它没有世界末日、英雄使命或复杂剧情，只给玩家一个自然走动的理由。团队也刻意减少强引导，没有巨大箭头和打断式教程，让玩家按自己的节奏发现世界。

**对我们的启示**：作品集世界也需要一个轻任务，而不是让 HR 一进来就迷路。可以把主线设计成"收集 3-5 个印章/名片/信件"，每到一个内容节点就获得一个清晰反馈。这样既有探索感，又能保证 5 分钟内看完主线。不要做复杂剧情，目标应是帮助 HR 高效理解你。

### 可访问性：非玩家也能自然上手

Messenger 使用越肩视角，并把操作降到极低门槛：手机上可以单指游玩，桌面上只用鼠标也能控制，智能相机会自动跟随玩家，避免用户频繁调整视角。团队提到，非游戏玩家也能不看说明直接开始玩。

**对我们的启示**：这是本项目能否真正运行起来的关键。HR 很可能不是游戏玩家，也可能在手机上打开链接。第一版应该避免复杂键鼠组合，至少提供：

- 桌面端：WASD + 鼠标拖拽，同时支持点击地面移动。
- 移动端：单指虚拟摇杆或拖拽移动，镜头自动跟随。
- 任何端：明显的"返回中心"和"打开简历 PDF/联系我"入口。

如果操作不顺，3D 形式会反过来伤害简历信息传达。

### 社交层：让世界显得活，但不要拥挤

Messenger 加入了多人层，但每个虚拟房间限制在 10 人左右，避免破坏安静探索的感觉。他们还放弃传统文字聊天，改用 3D emoji，降低沟通成本，也避免聊天审核和负面互动问题。

**对我们的启示**：个人作品集第一版不需要多人功能。可以用"伪社交"替代：少量 NPC、漂浮留言、自动播放的小提示、作品节点旁的访客脚印等，让世界有生命感即可。真正多人会带来服务器、同步、审核和性能成本，不适合作为 MVP。

### 声音设计：用区域声景引导探索

Messenger 为不同区域设计了不同 soundscape，例如城市噪声、海浪、工厂机械声。甚至有 NPC 弹吉他，玩家靠近时声音变大，声音本身成为自然导航的一部分。

**对我们的启示**：声音可以增强沉浸感，但不要作为第一版的硬依赖。MVP 可以先加轻量环境音和交互音效，并提供默认静音/一键开关。后续再为不同区域加声景，例如办公室键盘声、工坊机械声、山路风声、影院低频氛围声。

### 角色模型更新

实时更新角色模型通常会导致游戏卡顿。他们的方案是"滴漏式"更新 — 不一次性推送所有变化，而是分批次、平滑地将模型更新流入渲染管线。即使多个玩家同时更换服装，游戏也保持流畅。

**对我们的启示**：如果未来加入动态内容加载或多用户交互，数据推送应采用分步流式机制，避免集中爆发导致卡顿。

### 植被渲染

森林场景不是每棵树都精细建模。先用简单的"团块"（blob）占位，再通过几何数据与着色器的配合，自动替换为风格化的草木几何体。以极低的建模成本实现了好看的植被效果。

**对我们的启示**：户外探索径的山林场景可以采用相同策略 — 用简化几何体 + shader 做出风格化植被，不需要逐棵建模。

### UI 与文字渲染

UI 直接构建在 WebGL 之上，而非使用 HTML overlay。这样每个 UI 元素都能自由做动效，不受 DOM 限制。

文字渲染跳过了常规的 MSDF 字体方案，改用 WebAssembly 实时生成字形，由 GPU 直接渲染。即使文字处于极端角度，也能保持极为锐利的视觉效果。

**对我们的启示**：3D 场景内的文字内容（作品标题、描述、路标等）如果用纯 WebGL 渲染，效果比 HTML 叠加更好，但开发门槛更高。可作为进阶优化方向。

### 自定义 4 级 LOD 系统

为保持高帧率，实现了四档细节层次（Level of Detail）。玩家远离物体时自动切换为低精度模型，但切换过程中保留了物体的整体视觉轮廓，最大限度减少游戏中常见的"突跳"感。

LOD 0（最高精度）的基础网格展示了日式建筑和港口场景，网格密度高但拓扑规整，包含日文招牌、屋顶水箱、船、吊臂、人行桥等丰富细节。

**对我们的启示**：移动端运行时 LOD 几乎是必须做的。不同区域（实习大道 vs 户外探索径）的模型复杂度不同，需要统一规划 LOD 策略。

### 内存管理

Safari on iPhone 对内存极其敏感，超限后浏览器会直接杀掉页面，连警告都不会给。对策包括：追踪每一个资产，不用的立刻从内存清掉；使用自研工具对模型和纹理进行分割和压缩，在保持画质的同时控制内存占用。

**对我们的启示**：HR 很可能用手机打开链接，移动端内存管理必须提前规划。资产按需加载、及时释放，不能整个场景一次性载入。

### 颜色图集

整个游戏只用了一张 16×16 像素的微小纹理来存储所有颜色。模型不需要各自的大纹理贴图，而是从这个颜色查找表中取色。配合 Lo-fi 手绘风格，大幅减少显存占用。

**对我们的启示**：如果采用风格化/low-poly 美术方向，颜色图集是一个极致的优化手段。每个区域可以用不同的色板，既统一风格又节省资源。

---

### 技术哲学总结

这个团队的核心理念是：**用极致的工程优化来支撑极简的美术风格**。场景看着小、看着简单，但底层的渲染管线、内存管理、LOD 系统都是认真打磨的。

对我们的项目来说，关键启示是：不需要追求大场景和复杂美术，但工程侧的优化（尤其是移动端性能）要在项目初期就纳入规划，不能等做完了再补救。

### 对"个人作品集世界"的落地路线

为了让项目比较好地运行起来，建议不要一开始复刻 Messenger 的完整技术复杂度，而是采用"先跑通，再精修"的路线。

**MVP 阶段：能打开、能走、能看懂**

- 技术栈：Vite + React + Three.js / React Three Fiber。
- 场景范围：中央路口 + 1 个完整区域，例如"实习大道"。
- 核心交互：移动、自动跟随镜头、靠近节点弹出详情、返回中心、打开简历/联系方式。
- 美术策略：low-poly、统一色板、简单描边、少量重复资产。
- 内容策略：先放 3-5 个最能代表你的节点，不追求一次塞满所有经历。
- 性能目标：手机可打开，首屏加载尽量控制在几秒内；模型和图片按需加载。

**第二阶段：让世界更像作品，而不只是 demo**

- 增加 3-5 个主题区域，但每个区域都保持小体量。
- 为每个区域建立独立的色板、声音和交互形式。
- 做基础 LOD、纹理压缩和资源释放。
- 增加轻任务线，例如"收集个人名片碎片"或"完成一次作品路线"。
- 增加 NPC 或环境提示，让用户不用读说明也知道去哪里。

**第三阶段：向 Messenger 靠近的高级优化**

- 球形/小行星地形。
- 更完整的 LOD 系统。
- WebGL 内 UI 或 3D 字体优化。
- 更细致的声景和空间音频。
- 多区域资产流式加载与卸载。

优先级判断：**操作顺畅 > 信息清晰 > 风格统一 > 技术炫技**。这个项目的最终目标是让 HR 记住你、理解你，而不是证明你能做一个复杂游戏引擎。
