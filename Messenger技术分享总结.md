## Messenger (abeto.co) 原团队技术分享 — 要点总结

本文整理自 Messenger 原开发团队的技术分享，涵盖他们在 WebGL 浏览器游戏中所做的关键技术决策。这些方案对"个人作品集世界"项目有直接参考价值。

---

### 角色模型更新

实时更新角色模型通常会导致游戏卡顿。他们的方案是"滴漏式"更新 — 不一次性推送所有变化，而是分批次、平滑地将模型更新流入渲染管线。即使多个玩家同时更换服装，游戏也保持流畅。

**对我们的启示**：如果未来加入动态内容加载或多用户交互，数据推送应采用分步流式机制，避免集中爆发导致卡顿。

### 植被渲染

森林场景不是每棵树都精细建模。先用简单的"团块"（blob）占位，再通过几何数据与着色器的配合，自动替换为风格化的草木几何体。以极低的建模成本实现了好看的植被效果。

**对我们的启示**：户外探索径的山林场景可以采用相同策略 — 用简化几何体 + shader 做出风格化植被，不需要逐棵建模。

### UI 与文字渲染

UI 直接构建在 WebGL 之上，而非使用 HTML overlay。这样每个 UI 元素都能自由做动效，不受 DOM 限制。

文字渲染跳过了常规的 MSDF 字体方案，改用 WebAssembly 实时生成字形，由 GPU 直接渲染。即使文字处于极端角度，也能保持极为锐利的视觉效果。

**对我们的启示**：3D 场景内的文字内容（作品标题、描述、路标等）如果用纯 WebGL 渲染，效果比 HTML 叠加更好，但开发门槛更高。可作为进阶优化方向。

### 自定义 4 级 LOD 系统

为保持高帧率，实现了四档细节层次（Level of Detail）。玩家远离物体时自动切换为低精度模型，但切换过程中保留了物体的整体视觉轮廓，最大限度减少游戏中常见的"突跳"感。

LOD 0（最高精度）的基础网格展示了日式建筑和港口场景，网格密度高但拓扑规整，包含日文招牌、屋顶水箱、船、吊臂、人行桥等丰富细节。

**对我们的启示**：移动端运行时 LOD 几乎是必须做的。不同区域（实习大道 vs 户外探索径）的模型复杂度不同，需要统一规划 LOD 策略。

### 内存管理

Safari on iPhone 对内存极其敏感，超限后浏览器会直接杀掉页面，连警告都不会给。对策包括：追踪每一个资产，不用的立刻从内存清掉；使用自研工具对模型和纹理进行分割和压缩，在保持画质的同时控制内存占用。

**对我们的启示**：HR 很可能用手机打开链接，移动端内存管理必须提前规划。资产按需加载、及时释放，不能整个场景一次性载入。

### 颜色图集

整个游戏只用了一张 16×16 像素的微小纹理来存储所有颜色。模型不需要各自的大纹理贴图，而是从这个颜色查找表中取色。配合 Lo-fi 手绘风格，大幅减少显存占用。

**对我们的启示**：如果采用风格化/low-poly 美术方向，颜色图集是一个极致的优化手段。每个区域可以用不同的色板，既统一风格又节省资源。

---

### 技术哲学总结

这个团队的核心理念是：**用极致的工程优化来支撑极简的美术风格**。场景看着小、看着简单，但底层的渲染管线、内存管理、LOD 系统都是认真打磨的。

对我们的项目来说，关键启示是：不需要追求大场景和复杂美术，但工程侧的优化（尤其是移动端性能）要在项目初期就纳入规划，不能等做完了再补救。
