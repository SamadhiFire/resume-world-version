## Web3D 作品集资产执行规范总结

这一部分的核心目标是：**从建模第一步开始，就按照浏览器可稳定加载的标准执行**。不是先做复杂模型再优化，而是提前规定好上色、LOD、材质、碰撞、分包和加载规则。

---

# 一、核心原则

你的作品集是浏览器 Web3D 项目，所以资产制作必须遵守：

> **轻量、统一、可复用、可分级、可按需加载。**

具体来说：

```text
统一色板上色
低模模块化建模
四级 LOD
视觉模型与碰撞模型分离
GLB 分包加载
图片和视频不打进 3D 模型
持续在浏览器中测试
```

这套方式可以参考 Messenger 的技术思路：它不是靠高精度贴图和复杂材质取胜，而是通过风格化色彩、低资源资产、自定义 LOD 和资产管理保证浏览器端运行稳定。

---

# 二、上色方案：统一 Color Atlas

## 1. 不建议的做法

不要给每栋建筑单独做材质或贴图，例如：

```text
building_01_wall_material
building_02_roof_texture
building_03_window_texture
```

这样会导致：

* 材质数量膨胀；
* 贴图体积变大；
* Draw Call 增加；
* 后期统一调色困难；
* 浏览器加载压力变大。

---

## 2. 推荐做法

采用：

> **全项目共用一张 Color Atlas 色板贴图。**

也就是制作一张很小的色板，例如：

```text
16×16 px 或 32×32 px
```

每个小格是一种颜色。建筑、道路、水系、桥、NPC、道具都从这张色板中取色。

---

## 3. Color Atlas 的作用

| 作用     | 价值              |
| ------ | --------------- |
| 减少贴图体积 | 浏览器加载更轻         |
| 统一视觉风格 | 整个作品集更像一个完整世界   |
| 减少材质数量 | 降低渲染压力          |
| 方便整体调色 | 可快速调整白天、黄昏、夜晚氛围 |
| 适合低模风格 | 与名字街区的风格化表达匹配   |

---

## 4. 推荐色板分组

可以将色板提前规划成几个区域：

```text
道路 / 地砖 / 石材
墙体基础色
屋顶色
木材 / 金属
玻璃 / 屏幕
水体
植物
灯光 / 发光提示
阴影 / AO 色
NPC 服装
```

这样后续扩展建筑和 NPC 时，颜色不会混乱。

---

# 三、建筑上色执行方式

在 Blender 中：

```text
1. 所有建筑共用同一张 Color Atlas
2. 每个建筑面片通过 UV 指向色板中的颜色格
3. 墙面、屋顶、窗户、门、灯牌分别采样不同色块
4. 不使用独立高清贴图
5. 不使用复杂 PBR 材质
6. 只保留基础色和少量发光效果
```

例如：

```text
墙面 → 米白色块
屋顶 → 红棕色块
窗户 → 深蓝色块
门 → 木色块
交互灯牌 → 暖黄色发光色块
```

最终效果是：建筑有足够的视觉变化，但资源仍然非常轻。

---

# 四、材质控制规范

即使使用 Color Atlas，也要严格控制材质数量。

推荐全项目只保留少量核心材质：

```text
M_Atlas_Unlit        普通色板材质
M_Atlas_Lit          受简单光照影响的色板材质
M_Atlas_Emissive     发光提示材质
M_Water              简单水面材质
M_Sky                渐变天空材质
M_UI_Transparent     少量透明提示材质
```

重点是：

> **不要让每栋建筑拥有自己的独立材质系统。**

所有建筑尽量共用同一套材质，只通过 UV 采样不同色块形成差异。

---

# 五、四级 LOD 规范

## 1. LOD 是什么

LOD 即 **Level of Detail，细节层级**。
同一个建筑准备多个精度版本，根据玩家距离自动切换。

目的是：

```text
近处看细节
远处看轮廓
极远处只看剪影
```

---

## 2. 建议做 4 级 LOD

| LOD  | 使用距离      | 表现                  |
| ---- | --------- | ------------------- |
| LOD0 | 近距离       | 完整低模细节：门窗、招牌、屋檐、小装饰 |
| LOD1 | 中距离       | 保留整体形体，删除小装饰        |
| LOD2 | 远距离       | 只保留建筑大块体和屋顶轮廓       |
| LOD3 | 极远距离 / 总览 | 盒体剪影或合并建筑群          |

注意：
LOD0 也不是高模，而是 **浏览器友好的完整低模版本**。

---

## 3. 建筑面数参考

### 普通建筑

| 层级   |         面数建议 |
| ---- | -----------: |
| LOD0 | 300–800 tris |
| LOD1 | 150–300 tris |
| LOD2 |  40–120 tris |
| LOD3 |   12–40 tris |

### 地标建筑

| 层级   |           面数建议 |
| ---- | -------------: |
| LOD0 | 1000–2500 tris |
| LOD1 |  500–1000 tris |
| LOD2 |   120–300 tris |
| LOD3 |     30–80 tris |

### 小道具

| 层级   |            面数建议 |
| ---- | --------------: |
| LOD0 |     50–300 tris |
| LOD1 |      20–80 tris |
| LOD2 | 简化块 / billboard |
| LOD3 |              隐藏 |

---

# 六、LOD 制作方法

## 1. 重要建筑：手工制作 LOD

重要建筑不要完全依赖自动减面。

推荐流程：

```text
LOD0：完整低模建筑
LOD1：删除窗框、招牌厚度、小装饰
LOD2：合并门窗为色块，简化屋顶
LOD3：只保留建筑剪影和主色
```

这样轮廓和视觉质量最可控。

---

## 2. 普通建筑：自动减面 + 手动修正

可以用 Blender Decimate 辅助：

```text
复制 LOD0
↓
Decimate 减面
↓
手动修破面、法线、轮廓
↓
继续生成 LOD2
↓
LOD3 手工做成盒体剪影
```

自动减面只能辅助，不能完全依赖。

---

## 3. 远景建筑：合批成组

远处建筑不需要每栋独立 LOD。
可以把远景建筑群合并为一个整体：

```text
远景建筑群 LOD0：简化建筑组
远景建筑群 LOD1：更少盒体
远景建筑群 LOD2：整体剪影
```

这样可以减少 Draw Call 和模型数量。

---

# 七、LOD 切换注意事项

为了避免建筑在远近切换时突然“跳变”，需要注意：

```text
各级 LOD 外轮廓要尽量一致
各级 LOD 使用同一套 Color Atlas
切换距离设置缓冲区
不要在临界距离反复切换
重要建筑可做淡入淡出
远处保留主色和大形体即可
```

关键不是只减面，而是：

> **保持识别度和轮廓稳定。**

---

# 八、Color Atlas 与 LOD 的配合

LOD 不只是模型变简单，颜色也要同步简化。

| 层级   | 色彩处理          |
| ---- | ------------- |
| LOD0 | 保留门窗、招牌、细节色块  |
| LOD1 | 保留墙体、屋顶、窗户大色块 |
| LOD2 | 只保留墙体和屋顶主色    |
| LOD3 | 单色或双色剪影       |

所有 LOD 都必须共用同一张 Color Atlas。
不要让 LOD0 和 LOD1 使用不同材质或不同贴图。

---

# 九、碰撞模型必须独立

视觉模型不能直接作为碰撞模型。

每栋建筑需要拆成两类：

```text
视觉模型：给用户看的建筑，有外观和细节
碰撞模型：给角色和相机检测用的简化盒体
```

碰撞模型应该是：

```text
简单长方体
简单墙体
简单坡道
简单桥面
低面数地面
```

这样角色移动、NPC 巡逻、相机避障都会轻很多。

---

# 十、GLB 分包加载

不要把整个世界塞进一个巨大 GLB。
建议按功能拆分：

```text
map_base.glb              地形、主路、水系基础
buildings_core_lod.glb    核心建筑 + LOD
buildings_far_lod.glb     远景建筑群 + LOD
props_common.glb          树、路灯、长椅、小物件
bridges_water.glb         桥、水渠、水面
interaction_markers.glb   发光点、提示牌、入口标识
player.glb                玩家角色
npc_common.glb            通用 NPC
collision_world.glb       碰撞模型
```

加载顺序建议：

```text
第一批：天空、地面、主街道、角色、入口建筑
第二批：核心建筑 LOD、NPC、交互点
第三批：远景建筑、装饰、氛围 NPC
第四批：用户点击后加载作品图片 / 视频 / 链接内容
```

---

# 十一、图片和视频不要打进 3D 模型

你的作品集中会有：

```text
AIGC 视频
GitHub 链接
文章链接
作品截图
简历 PDF
户外照片
```

这些都不要放进 GLB。

推荐方式：

```text
3D 场景中只放入口、NPC、路牌、缩略图占位
用户点击后用 HTML 面板加载真实图片、视频或链接
```

这样可以显著降低首屏加载压力。

---

# 十二、模块化建筑资产库

高质量不等于每栋建筑都重新建。
更好的方式是建立一套模块库：

```text
墙体模块
屋顶模块
门窗模块
招牌模块
阳台模块
楼梯模块
桥模块
水渠模块
围栏模块
路灯模块
树木模块
长椅模块
摊位模块
```

通过组合生成多样建筑。

这样可以同时实现：

```text
视觉丰富
风格统一
资源复用
模型轻量
后期易维护
```

---

# 十三、Draw Call 控制

WebGL 里不只是面数重要，Draw Call、材质数量、透明物体和实时阴影也很重要。

需要控制：

```text
材质数量
mesh 数量
透明材质数量
动态光源数量
实时阴影数量
骨骼动画数量
视频纹理数量
```

建议：

```text
全项目核心材质尽量少于 10 个
普通建筑尽量共用 1 个 Atlas 材质
重复道具可实例化
透明材质少用
实时点光源少用
实时阴影少用
视频不要作为场景纹理常驻
```

---

# 十四、BlenderMCP 的正确使用方式

BlenderMCP 可以用，但不能让它自由生成最终资产。

它适合用于：

```text
生成建筑草模
批量生成街道装饰
快速做桥、路牌、摊位、小道具
辅助调整场景布局
```

但正式资产必须经过：

```text
比例统一
面数检查
法线检查
命名规范
Color Atlas 上色
LOD 分级
碰撞体制作
GLB 导出
浏览器测试
```

最终流程应该是：

```text
BlenderMCP 生成草模
↓
Blender 手动清理
↓
按 Color Atlas 上色
↓
制作 4 级 LOD
↓
制作简化碰撞体
↓
导出 GLB
↓
压缩优化
↓
浏览器测试
```

---

# 十五、正式资产规范总结

可以把你的资产标准概括为：

> 所有 3D 资产必须遵循 WebGL 浏览器优先原则。建筑、道路、桥梁、道具统一采用低多边形风格，并使用全项目共享 Color Atlas 上色。不得为单栋建筑创建独立高清贴图或复杂材质。所有主要建筑必须提供 LOD0、LOD1、LOD2、LOD3 四级模型，LOD 之间保持相近轮廓和一致色彩。视觉模型与碰撞模型分离，碰撞体必须使用简化几何体。图片、视频、文章封面等内容型资源不得打入 GLB，必须通过 HTML UI 或点击后懒加载方式呈现。模型导出后统一使用 GLB 格式，并进行压缩、体积检查和浏览器性能测试。

---

# 十六、一句话总结

> 你的 3D 名字街区作品集应该从一开始就采用 Color Atlas 上色、四级 LOD、模块化建筑、简化碰撞、GLB 分包、内容资源懒加载的正式资产管线。这样既能保留丰富的 3D 街区表现，又能保证电脑浏览器稳定打开和流畅运行。
