# CHEN YANG 3D 作品集地图：产品设计技术路线图

## 一、产品定位

### 1. 产品名称

暂定：

> **CHEN YANG Walkable Portfolio Map**
> **陈炀的 3D 名字街区作品集**

### 2. 一句话概括

> 将 “CHEN YANG” 设计成一张可行走的 3D 平面地图。用户从高空俯视能看出你的名字，进入地面后则以第三人称角色在街道、建筑、小桥、水系和 NPC 之间探索，通过对话和链接了解你的实习经历、Vibe Coding 作品、AIGC 内容、兴趣爱好和个人档案。

### 3. 核心体验目标

这个作品集不是传统网页，也不是大型游戏，而是一个 **轻量化 Web3D 个人能力小镇**。

它的目标有四个：

| 目标    | 说明                      |
| ----- | ----------------------- |
| 记忆点   | 俯视时地图构成 “CHEN YANG”     |
| 探索感   | 用户能操控角色在街区中行走           |
| 信息清晰  | 内容通过 NPC 对话、链接卡片和 UI 展示 |
| 浏览器稳定 | 电脑端打开顺畅，不依赖下载或安装        |

---

# 二、核心设计原则

## 1. 不做球形，做平面 3D 名字地图

你的项目不需要像 Messenger 一样做球形星球。球形会增加相机、重力、地形、移动方向和建模难度。你的目标是作品集展示，所以更适合：

> **平面地图 + 3D 街区 + 第三人称角色行走**

从高空看，是你的名字；从地面看，是一个自然的小镇。

---

## 2. 名字只作为宏观轮廓，不绑定内容模块

“CHEN YANG” 不需要每个字母对应一个固定功能区。
更好的方式是：

> **名字负责俯视识别，内部空间自由设计。**

也就是说，地图可以用主路、水系、建筑轮廓、绿化和灯带共同构成字母，但内部建筑、NPC、桥、坡道、节点可以根据体验自由安排。

---

## 3. 采用“混合构字方案”

地图不要只用道路硬拼字母，也不要只靠建筑构字。最适合你的方案是：

> **主路形成字母骨架，建筑、水系、绿化、灯带、桥和留白补充字母识别。**

例如：

| 字母结构  | 可用空间元素     |
| ----- | ---------- |
| C 的弧形 | 道路、水池、树木留白 |
| H 的横竖 | 建筑街区、桥、广场  |
| E 的三横 | 小巷、灯带、低矮建筑 |
| N 的斜线 | 水渠、斜坡路、小桥  |
| Y 的分岔 | 分岔步道、河流分支  |
| A 的三角 | 三角广场、屋顶轮廓  |
| G 的回环 | 环形花园、回廊、水岸 |

核心效果是：

> **宏观看得出名字，微观看起来像自然街区。**

---

## 4. 3D 负责空间感，HTML UI 负责阅读

不要把大量文字直接贴在 3D 场景里。
3D 场景负责：

* 角色行走；
* 街道探索；
* 建筑氛围；
* NPC 互动；
* 视觉记忆点。

详细信息用 HTML/CSS 面板承载：

* 对话框；
* 链接卡片；
* 项目介绍；
* GitHub 链接；
* 视频链接；
* 简历下载；
* 联系方式。

这样既保留游戏感，又不会牺牲信息清晰度。

# 三、推荐技术栈

## 1. 前端框架

推荐：

```text
Vite + React + TypeScript
```

如果你后续想做普通网页版、SEO、项目详情页，也可以用：

```text
Next.js + React + TypeScript
```

第一版为了开发轻快，我更建议 **Vite + React**。

---

## 2. 3D 渲染

推荐：

```text
Three.js
React Three Fiber
Drei
```

React Three Fiber 是 Three.js 的 React renderer，适合把 3D 场景拆成可复用组件，并与 React 状态和 UI 体系结合。([Poimandres 文档][2])

---

## 3. 状态管理

推荐：

```text
Zustand
```

用于管理：

* 当前角色位置；
* 当前 NPC；
* 当前对话内容；
* 已访问节点；
* 小地图状态；
* 快速导览进度；
* 是否打开总览视角；
* 是否打开简历/链接卡片。

---

## 4. 3D 资产格式

推荐：

```text
GLB / glTF
```

glTF 是面向 3D 场景和模型的高效传输格式，目标就是降低 3D 资产大小和运行时处理成本，适合 Web 端实时 3D 项目。([The Khronos Group][3])

---

## 5. 建模与资源工具

建议工具链：

```text
Figma：画俯视地图草图
Blender：建模、清理、导出 GLB
Meshy / BlenderMCP：辅助生成简单资产
glTF Transform：压缩和优化 GLB
Photoshop / Photopea：图片处理
WebP / AVIF：图片压缩格式
```

AI 生成模型不能直接放进网页，必须经过 Blender 清理、减面、合并材质和压缩。

# 五、地图系统设计

## 1. 地图形态

整体是：

```text
平面 3D 地图
+
俯视可读的 CHEN YANG
+
地面可行走的街区小镇
```

用户进入后可以看到：

```text
CHEN
YANG
```

但地面行走时不应该像在字体笔画里走，而应该像在一座有道路、建筑、水系和 NPC 的微型城市中探索。

---

## 2. 地图组成

地图可以由这些元素构成：

| 元素 | 作用                 |
| -- | ------------------ |
| 主路 | 形成 CHEN YANG 的主要笔画 |
| 支路 | 连接建筑、NPC 和互动节点     |
| 水系 | 增加层次，也能辅助构字        |
| 小桥 | 连接不同街区，增强空间变化      |
| 建筑 | 制造 3D 体积和街区感       |
| 绿化 | 柔化字母边缘             |
| 灯带 | 夜间或俯视时强化字母轮廓       |
| 广场 | 作为停留点、转折点和对话点      |
| 路牌 | 引导用户理解方向           |

---

## 3. 地图制作流程

建议流程：

```text
Figma 画 CHEN YANG 俯视草图
↓
确定主路、水系、建筑大致位置
↓
导入 Blender 作为参考图
↓
在 Blender 中搭建低模地形和街道
↓
加入建筑块、小桥、水渠、绿化
↓
导出 GLB
↓
在 R3F / Three.js 中加载测试
↓
根据相机视角和行走体验调整比例
```

第一版不要程序化生成地图，先手工搭一个可控原型。

# 六、角色移动系统

## 1. 视角形式

采用：

> **第三人称背后跟随视角**

用户能看到角色全身，相机在角色后上方，略微俯视。

这比第一人称更适合作品集，因为：

* 不容易晕；
* 能看到角色和环境；
* 更有游戏感；
* 方便理解空间；
* 不会遮挡大量 UI。

---

## 2. 操作方式

建议支持：

```text
WASD / 方向键：移动
Shift：可选加速
鼠标：轻微调整视角
E：与 NPC 交互
Esc：关闭对话
M：打开小地图 / 总览
```

同时要保留非游戏化操作：

```text
顶部导航跳转
快速导览按钮
回到入口按钮
打开总览按钮
下载简历按钮
```

因为 HR 不一定愿意慢慢操作角色。

---

## 3. 上坡下坡处理

地形可以有轻微坡度、小桥、台阶，但不要太复杂。
角色高度可以通过 **Raycast 贴地** 实现：

```text
角色移动到新 x/z 坐标
↓
从角色上方向下发射射线
↓
检测脚下地面高度
↓
更新角色 y 坐标
```

这样角色能自然走上坡、下坡和小桥。

---

# 七、第三人称相机系统

## 1. 基础相机

相机位置：

```text
角色后方
角色上方
略微俯视
```

目标是：

* 能看到角色全身；
* 能看到前方街道；
* 画面上方有天空；
* 不要看到太远的地图边界。

---

## 2. 相机避障

街道里会有建筑和坡道，所以相机可能穿墙。
需要做简单避障：

```text
从角色到相机位置发射射线
↓
如果中间有建筑遮挡
↓
相机自动拉近
↓
遮挡消失后相机平滑恢复
```

Messenger 官方复盘也强调了动态相机的重要性，尤其是在复杂街巷和球形世界中避免相机不稳定；你的项目虽然是平面地图，但第三人称相机同样需要基础避障。([Awwwards][1])

---

## 3. 总览相机

需要一个高空俯视按钮：

```text
点击“总览”
↓
相机升高
↓
显示完整 CHEN YANG 地图
↓
标出当前位置
```

这是你作品集最重要的记忆点展示。

---

# 八、天空与远景系统

## 1. 天空方案

推荐：

> **渐变 Skydome + 远景雾 + 少量低模云**

Three.js 官方 `Sky` 代表一种 skydome 场景背景方案；你的项目不一定要用真实感天空模型，可以用更轻量的渐变天空穹顶实现类似“包裹场景”的效果。([threejs.org][4])

---

## 2. 天空层原则

天空只负责氛围，不参与交互。

```text
不碰撞
不投影
不接收阴影
不加载大图
不做复杂天气
不参与物理
```

推荐做法：

```text
一个巨大半球 / 球体
渐变材质
少量云朵 sprite 或低模云
可跟随相机位置
```

---

## 3. 地图边界处理

因为你的地图是平面，角色上坡时可能看到地图边缘。
解决方式：

```text
地图外圈加低模山体
加树林
加建筑剪影
加雾气
加水岸或围栏
```

这样用户不会看到地图尽头。

# 九、NPC 系统

## 1. NPC 类型

分三类：

| 类型     | 用途              |
| ------ | --------------- |
| 固定 NPC | 导览、简历、联系、核心内容入口 |
| 巡逻 NPC | 按路线行走，让街区有生命感   |
| 氛围 NPC | 不一定可对话，只增加世界活力  |

第一版建议控制数量：

```text
5–8 个固定 NPC
2–3 个巡逻 NPC
少量氛围 NPC
```

不要一开始做 25 个以上 NPC。

---

## 2. 巡逻实现方式

采用：

> **Waypoint 固定路线 + Raycast 贴地 + 距离触发对话**

逻辑：

```text
NPC 读取路线点
↓
沿路线点移动
↓
Raycast 自动贴合地面高度
↓
移动时播放 Walk 动画
↓
玩家靠近后出现“点击对话”
↓
点击后 NPC 停止、面向玩家
↓
打开对话框
↓
对话结束后继续巡逻
```

---

## 3. NPC 数据结构

NPC 内容不要写死在代码里，用 JSON 配置：

```ts
{
  id: "guide-profile",
  name: "导览员",
  type: "fixed",
  position: [2, 0, 4],
  dialogue: [
    {
      speaker: "导览员",
      text: "欢迎来到 CHEN YANG Walkable Portfolio Map。"
    },
    {
      speaker: "导览员",
      text: "你可以先看一份快速简历。",
      link: {
        label: "下载 PDF 简历",
        url: "/resume.pdf"
      }
    }
  ]
}
```

这样后续新增内容时，只需要增加配置，不需要改核心系统。

# 十、对话与链接系统

## 1. 对话形式

建议用：

```text
聊天气泡
+
选项按钮
+
链接卡片
```

例如：

```text
NPC：这里展示的是陈炀的 Vibe Coding 项目。
玩家可选：
- 看代表项目
- 打开 GitHub
- 看视频演示
- 下次再说
```

---

## 2. 链接卡片

不要裸露长链接，做成卡片：

```text
[GitHub 项目]
一个用于自动化工作流的工具
按钮：打开链接
```

可支持：

* GitHub；
* AIGC 视频；
* 文章链接；
* 项目 Demo；
* 简历下载；
* 作品详情页；
* 邮箱联系。

---

## 3. 信息层级

每个 NPC 对话不要太长。
建议三层结构：

```text
第一层：一句话介绍
第二层：关键经历 / 作品说明
第三层：链接或详情卡片
```

这样 HR 可以快速扫，也可以选择深入看。

---

# 十一、内容数据系统

你的内容应该和 3D 场景分离。

建议结构：

```text
/content
  profile.json
  internships.json
  vibe-coding.json
  aigc.json
  hobbies.json
  articles.json
  links.json
  npcs.json
```

3D 场景只管：

* NPC 在哪里；
* 建筑是什么样；
* 路牌在哪里；
* 触发点在哪里。

内容由 JSON 管理。
这样后续更新实习、项目、链接时，不需要重新建模。

# 十二、加载与性能路线

## 1. 加载顺序

推荐加载顺序：

```text
1. HTML 首页和基础 UI
2. 轻量天空背景
3. 基础地面和入口街区
4. 主角模型
5. CHEN YANG 主街道
6. 附近建筑和 NPC
7. 远景、云、树、装饰
8. 视频、图片、GitHub、作品内容点击后再加载
```

重点是：

> **用户打开后先看到内容，不要等完整 3D 世界加载完才显示。**

---

## 2. 模型拆分

不要把所有东西打成一个巨大 GLB。

建议拆分：

```text
map-base.glb
roads.glb
water-bridges.glb
buildings-core.glb
buildings-deco.glb
props.glb
npcs.glb
character.glb
```

如果后期地图变大，可以按区域继续拆分。

---

## 3. 资源预算

建议第一版预算：

| 类型       | 建议                    |
| -------- | --------------------- |
| 首屏 3D 资源 | 5 MB 左右               |
| 全站 3D 资源 | 20 MB 以内              |
| 单个 NPC   | 低模、少骨骼、少贴图            |
| 贴图       | WebP / 小尺寸 / 尽量复用     |
| 视频       | 不进入首屏                 |
| 帧率       | 主流电脑 60fps，低配至少 30fps |

---

## 4. 模型优化流程

```text
Blender 建模
↓
减面
↓
合并材质
↓
导出 GLB
↓
glTF Transform 压缩
↓
浏览器中测试加载和帧率
```

glTF Transform 可用于优化 glTF / GLB 资产，Draco 压缩可以减少静态几何体数据体积，但前提仍然是模型本身要低面数、材质数量少。([GitHub][5])

---

## 5. 降级策略

如果设备性能较低，可以自动关闭：

```text
动态云
复杂阴影
远景装饰
部分 NPC 动画
后处理效果
高质量描边
```

低配模式可以只保留：

```text
基础天空
主街道
角色
关键 NPC
基础建筑
HTML 信息面板
```

# 十三、普通网页兜底

必须做一个普通网页版。

原因是：

* 有些浏览器 WebGL 表现不稳定；
* 有些 HR 没时间走完整地图；
* 有些人只想看简历；
* 求职场景需要兜底。

普通网页版内容包括：

```text
个人介绍
教育背景
实习经历
Vibe Coding 项目
AIGC 作品
文章链接
兴趣爱好
简历下载
联系方式
```

3D 版负责记忆点，普通版负责稳妥阅读。
